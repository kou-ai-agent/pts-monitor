# PTS株価監視システム 基本設計仕様書

**作成日**：2026年3月20日  
**ステータス**：確定（Antigravity開発依頼用）

---

## 1. プロジェクト概要

### 目的
PTSの株価動きを日々自動収集し、外出先のスマホからでも閲覧できる個人用Webページを構築する。

### 性格づけ
単純な自動化（RPA）ではなく、**AIエージェントとして設計する**。Claude APIによる判断・推論・生成を組み込み、データを読んで意味を生み出すシステムとする。

---

## 2. データ要件

### 取得種別（4種類）
| 種別 | 内容 |
|---|---|
| 値上がり率 | PTS終値ベースの騰落率上位 |
| 値下がり率 | PTS終値ベースの騰落率下位 |
| 出来高上位 | 取引量上位 |
| 売買代金上位 | 取引金額上位 |

### 取得市場（4区分）
- 全市場
- プライム
- スタンダード
- グロース

### 取得件数
各種別×各市場で**100位まで**

### データソース
**株探（kabutan.jp）スクレイピング**

- 利用規約：グレーゾーンと認識した上で、個人・非公開・日次1回・低頻度という条件で割り切って採用
- 将来：三菱UFJ eスマート証券（旧kabuステーション）が個人向けAPIを開放した時点で差し替える前提で、**データ取得層を分離設計すること**（後述）
- robots.txt を必ず確認・遵守すること

### 初期データ
- システム稼働初日に**過去30日分を一括スクレイピングで取得**する
- リクエスト間隔は最低1秒以上空けること
- 初期取得スクリプトは本番の定期取得スクリプトと共通化すること

## 3. AIエージェント機能

Claude API（claude-haiku-4-5）を使用。日次2回のAPIコールで以下を生成する。

### ① 注目銘柄の自動選定・コメント生成
**インプット**
- 当日の全ランキングデータ（4種別）
- 対象銘柄の過去30日間のランキング履歴

**処理**
- 注目すべき銘柄を最大5つ選定
- 選定理由を各2〜3文で生成
- 判断基準の例：連続ランクイン・急浮上・出来高急増・複数種別での同時ランクイン

**アウトプット形式（JSON）**
```json
[
  {
    "code": "1234",
    "name": "○○○○",
    "reason": "3日連続で値上がり率上位にランクイン。本日は出来高も急増しており...",
    "rank_today": 2,
    "category": "price_up"
  }
]
```

### ③ 当日PTSサマリーの自然言語生成
**インプット**
- 当日の値上がり・値下がり上位10銘柄
- 出来高・売買代金上位10銘柄

**処理**
- 当日のPTS全体の特徴を3〜4文で生成
- テーマ・セクター傾向・特異な動きがあれば言及

**アウトプット形式**
```
"本日のPTSはグロース銘柄に買いが集中。AIテーマ関連が値上がり率上位を占め、
特に○○セクターで出来高が急増した。一方、大型株は全般的に売りが先行..."
```

## 4. データ保存設計

### 形式
**JSON（SQLiteは使用しない）**

- SQLiteバイナリのGit管理を避けるためテキスト形式を採用
- 差分確認・デバッグが容易

### 保存ルール
- 過去**30日分**のみリポジトリで管理（それ以前は削除）
- ファイル名：`data/YYYY-MM-DD.json`
- インデックスファイル：`data/index.json`（存在する日付の一覧）

### 日次JSONの構造
```json
{
  "date": "2026-03-20",
  "generated_at": "06:30:00",
  "ai_summary": "本日のPTSは...",
  "ai_highlights": [

      "code": "1234",
      "name": "○○○○",
      "reason": "...",
      "rank_today": 2,
      "category": "price_up"
    }
  ],
  "rankings": {
    "price_up": {
      "all": [
        {
          "rank": 1,
          "code": "1234",
          "name": "○○○○",
          "price": 1234,
          "change_pct": 15.2,
          "volume": 100000,
```

## 5. Webページ機能要件

### 構成
**SPA（Single Page Application）構成**

- Python側はJSONを生成するのみ
- フロントエンドはHTML + Vanilla JS（またはAlpine.js等の軽量フレームワーク）でJSONを読み込んで描画
- Jinja2等のサーバーサイドテンプレートは使用しない

### 画面構成

```
┌─────────────────────────────────────────┐
│  PTS ランキングモニター                   │
│                                         │
│  📊 今日のサマリー（AI生成）              │
│  「本日のPTSはグロース銘柄に買いが集中。  │
│   AIテーマ関連が上位を占め…」            │
│                                         │
│  ⭐ 注目銘柄（AI選定・最大5件）           │
│  ┌──────────────────────────────┐      │
│  │ ○○○○（1234）                │      │
│  │ 「3日連続で値上がり率上位…」  │      │
│  └──────────────────────────────┘      │
│                                         │
│  [カレンダー: 2026/03/20 ▼]            │
│  [市場: 全市場 ▼]  [種別: 値上がり率 ▼] │
│                                         │
│  順位 │ 銘柄名   │ 価格    │ 変化率 │ 前日順位 │
│    1  │ ○○○○  │ 1,234円 │ +15.2% │   3位   │
│    2  │ △△△△  │   567円 │ +12.8% │  新規   │
│   ... │（100位まで）                    │
│                                         │
│  ※銘柄行をタップ → ポップアップ表示     │
│  ┌──────────────────────────┐          │
│  │ ○○○○ 過去30日 順位推移  │          │
│  │ [折れ線グラフ Chart.js]   │          │
│  └──────────────────────────┘          │
└─────────────────────────────────────────┘
```

### UI要件
- **スマホファースト**でデザインすること
- カレンダーで日付選択（データが存在する日付のみ選択可）
- 市場・種別はドロップダウンで切り替え（ページ再読み込みなし）
- ランキングテーブルは100位まで表示
- 前日順位の表示：数値、もしくは「新規」（前日圏外の場合）
- 銘柄行タップでポップアップ表示
  - 銘柄名・コード
  - 直近30日間の順位推移（折れ線グラフ）
  - グラフライブラリ：Chart.js（CDN経由）
  - 圏外の日は「101位」としてプロット

## 6. システム構成・ファイル構成

### アーキテクチャ
```
[株探 PTSランキングページ]
        ↓ スクレイピング（毎日 6:30 JST）
[scraper.py] ← データ取得層（将来API差し替えポイント）
        ↓
[processor.py] ← データ処理・JSON出力層
        ↓
[data/YYYY-MM-DD.json] ← 日次データ（30日分管理）
[data/index.json]      ← 日付インデックス
        ↓
[agent.py] ← Claude API連携（AIエージェント層）
        ↓ JSONに ai_summary, ai_highlights を追記
[main.py] ← 全体の実行制御・エラーハンドリング
        ↓
[GitHub Actions] ← 毎朝6:30に自動実行・commit・push
        ↓
[docs/index.html] ← SPA本体（GitHub Pages公開）
        ↓
[Cloudflare Access] ← 認証（自分のみアクセス可）
        ↓
[スマホ・外出先から閲覧]
```

### ファイル構成
```
project/
├── main.py                  # 全体実行制御
├── scraper.py               # データ取得層（株探スクレイピング）
├── processor.py             # データ処理・JSON出力
├── agent.py                 # Claude API連携
├── requirements.txt         # Python依存パッケージ
├── .github/
│   └── workflows/
│       └── daily.yml        # GitHub Actions設定
├── data/
│   ├── index.json           # 存在する日付の一覧
│   ├── 2026-03-20.json      # 日次データ
│   └── ...（30日分）
└── docs/
    ├── index.html           # SPA本体（GitHub Pages公開ルート）
    └── assets/
        ├── app.js           # フロントエンドロジック
        └── style.css        # スタイル
```

### レイヤー分離の設計方針
**scraper.pyは将来のAPI差し替えを前提に、インターフェースを明確に定義すること。**

```python
# scraper.py が返す標準フォーマット（processor.py はこれだけを受け取る）
def fetch_ranking(date: str, category: str, market: str) -> list[dict]:
    """
    Returns:
        [
            {
                "rank": int,
                "code": str,
                "name": str,
                "price": float,
                "change_pct": float,
                "volume": int,
                "turnover": int
            },
            ...
        ]
    """
```

このインターフェースを守れば、将来 scraper.py の中身をAPIコールに差し替えるだけで動作する。

## 7. 自動実行（GitHub Actions）

### daily.yml の要件
- **実行時刻**：毎日 21:30 UTC（= 日本時間 6:30 JST）
- **実行内容**：
  1. `main.py` を実行（スクレイピング → JSON保存 → Claude API → index.json更新）
  2. 30日より古いJSONファイルを削除
  3. `data/` 以下の変更をcommit・push
- **シークレット**：
  - `ANTHROPIC_API_KEY`：Claude API キー
- **エラー時**：ステップ失敗をGitHub上で確認できるようにする

---

## 8. ホスティング・アクセス制限

### GitHub Pages
- 公開ディレクトリ：`docs/`
- カスタムドメイン：不要（github.io のデフォルトURLで可）

### Cloudflare Access（自分だけアクセス可能にする）
- 無料プラン（Zero Trust Free）を使用
- 認証方式：メールOTP（ワンタイムパスワード）
- 設定手順は開発完了後に別途実施
